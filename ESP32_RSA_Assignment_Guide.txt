ESP32 RSA OPTIMIZATION ASSIGNMENT GUIDE
=======================================

Purpose
-------

This assignment teaches modular arithmetic, embedded C, benchmarking, and
correctness-first optimization.

Students are given a correct but slow RSA-style encryption function. They must
make the student implementation faster while keeping the same output.


1. RSA Operation
----------------

RSA encryption uses modular exponentiation:

    ciphertext = plaintext^e mod n

Where:

    plaintext   is the input integer
    e           is the public exponent
    n           is the modulus
    ciphertext  is the result

This assignment uses small toy values that fit in uint64_t. They are useful
for testing and learning, but they are not secure.


2. Starter Algorithm
--------------------

The starter code uses square-and-multiply:

    result = 1
    base = plaintext mod n

    while e > 0:
        if e is odd:
            result = result * base mod n
        base = base * base mod n
        e = e / 2

    return result

The starter multiplication avoids overflow by using repeated addition and
doubling. This is correct, but slow.


3. Files
--------

Important files:

    main/rsa_baseline.c   fixed baseline, do not change
    main/rsa_student.c    student implementation, optimize this
    main/app_main.c       tests and benchmark driver
    tools/score_host.py   quick host-side checker
    scripts/qemu_run.sh   QEMU helper script

The public function to optimize is:

    uint64_t rsa_encrypt_u64_student(uint64_t plaintext,
                                     uint64_t exponent,
                                     uint64_t modulus);

Do not change this signature.


4. Build and Test
-----------------

The project supports ESP-IDF, Docker, QEMU, and a host-side checker.

Students should inspect the repository and tool documentation to determine the
exact commands.

A successful run should show:

    TEST ... PASS
    BENCH_BASELINE ...
    BENCH_STUDENT ...
    RESULT correctness=PASS
    SCORE correctness_points=... speed_points=... total_points=...


5. Correctness
--------------

Correctness is required before performance counts.

A correct solution must:

    1. Match the expected ciphertext
    2. Work for hidden test vectors
    3. Be deterministic
    4. Keep the baseline unchanged
    5. Avoid undefined behavior
    6. Handle edge cases such as plaintext >= n, e = 1, and small moduli

Students must not hard-code the public answers.


6. Benchmarking
---------------

Good benchmarking rules:

    1. Run many iterations
    2. Do not print inside the timed loop
    3. Use a checksum so the compiler cannot remove the work
    4. Compare against the baseline
    5. Repeat measurements when possible

Speedup formula:

    speedup_percent = ((baseline_time - optimized_time) / baseline_time) * 100

The final judge may use:

    total_time_us = public_time_us + hidden_time_us

Lower time is better, but only if correctness passes.


7. Optimization Ideas
---------------------

Possible improvements, from simpler to harder:

    1. Remove unnecessary modulo operations
    2. Reduce repeated work
    3. Reduce branches in hot loops
    4. Use faster modular multiplication
    5. Precompute reusable values
    6. Try fixed-window exponentiation
    7. Try Montgomery multiplication
    8. Measure stack and heap use
    9. Tune compiler settings carefully

Students should make one change at a time, test correctness, then measure
again.


8. Embedded Measurement Notes
-----------------------------

Embedded timing can be noisy.

Common mistakes:

    1. Measuring serial print time
    2. Measuring setup instead of computation
    3. Running too few iterations
    4. Comparing different environments without explanation
    5. Forgetting compiler optimization settings

Reports should state the environment used.


9. Security Notes
-----------------

This assignment is not production cryptography.

Real RSA needs:

    1. Large keys
    2. Padding such as OAEP
    3. Reviewed cryptographic libraries
    4. Side-channel protection
    5. Careful memory handling

For this assignment, students may discuss timing side-channel tradeoffs, but
correctness and measured performance are the main goals.


10. Scoring Rubric
------------------

The score is based only on correctness and speed.

A submission must build and run in the assigned environment. A submission that
does not build receives 0 points.

1. Correctness, 60 points

    25 pts: all public tests pass
    20 pts: all hidden tests pass
    10 pts: edge cases work, including plaintext >= n, e = 1, and small moduli
    5 pts: repeated runs produce the same results

If any correctness test fails, the speed score is 0.

2. Speed, 40 points

Compute total time:

    baseline_total_us = public_baseline_us + hidden_baseline_us
    student_total_us  = public_student_us  + hidden_student_us

Then compute speedup:

    speedup = baseline_total_us / student_total_us

Then compute speed points:

    speed_points = min(40, max(0, 20 * (speedup - 1)))

Examples:

    same speed as baseline: speedup = 1.0, speed score = 0
    2x faster than baseline: speedup = 2.0, speed score = 20
    3x faster than baseline: speedup = 3.0, speed score = 40
    slower than baseline: speed score = 0

If any correctness test fails, the speed score is 0.


11. Student Report
------------------

The report should include:

    1. Name and environment
    2. Baseline benchmark
    3. Optimization description
    4. Optimized benchmark
    5. Speedup percentage
    6. Correctness output
    7. Tradeoffs or limitations


12. Summary
-----------

The optimization process is:

    Measure first.
    Change one thing.
    Test correctness.
    Measure again.
    Explain the result.
