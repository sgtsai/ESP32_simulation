ESP32 RSA HARDWARE ACCELERATOR NOTE
===================================

Purpose
-------

This note explains how the ESP32 RSA hardware accelerator fits into the RSA
optimization assignment.

The ESP32 includes hardware support for large-number cryptographic operations.
This hardware is commonly described as the RSA or MPI/bignum accelerator. It is
useful for operations such as:

    ciphertext = plaintext^e mod n

This is exactly the core mathematical operation used by RSA encryption.


1. Is It a CPU Instruction?
--------------------------

No. The ESP32 RSA accelerator is not a normal CPU instruction like:

    rsa_encrypt x, y, z

Instead, it is a hardware peripheral. Software writes large-number operands
into special hardware-controlled memory/register areas, starts the operation,
waits for completion, and reads the result back.

So the accelerator is more like a small cryptographic co-processor inside the
chip.


2. How Students Should Use It
-----------------------------

For this assignment, students should use the accelerator through ESP-IDF and
mbedTLS when possible, not by manually writing raw hardware registers.

Recommended path:

    mbedtls_mpi_exp_mod()

This function computes:

    X^E mod N

When ESP-IDF is configured with hardware MPI/RSA acceleration enabled, supported
big-number operations can be routed to the ESP32 hardware accelerator.

The repository includes:

    sdkconfig.defaults

with:

    CONFIG_MBEDTLS_HARDWARE_MPI=y

This setting is intended to enable ESP-IDF's hardware MPI acceleration support
for ESP32 targets that provide it.


3. Why Use mbedTLS Instead of Raw Registers?
--------------------------------------------

Using mbedTLS is better for this assignment because:

    1. It is easier for students to understand.
    2. It avoids fragile low-level register code.
    3. It follows ESP-IDF's normal software stack.
    4. It is more portable across ESP-IDF versions.
    5. It lets students focus on measurement and comparison.

Raw register programming is possible, but it is harder to teach and easier to
break. It is better as an advanced bonus topic.


4. What Should Be Compared?
---------------------------

Students should compare at least two paths:

    1. Software RSA toy implementation
    2. Hardware-accelerated mbedTLS MPI implementation

The software path shows how modular exponentiation works internally.

The hardware path shows how real embedded systems often accelerate expensive
cryptographic math.

Students should report:

    1. Software baseline time
    2. Hardware-accelerated time
    3. Correctness result for both paths
    4. ESP-IDF version
    5. Target chip
    6. Whether the test ran on real ESP32 hardware, QEMU, or host mock


5. QEMU Warning
---------------

QEMU is useful for build and correctness testing, but it should not be treated
as a perfect model of ESP32 hardware acceleration.

Important:

    QEMU timing is not real ESP32 timing.

If students want to claim hardware accelerator speedup, they should measure on
a real ESP32 board whenever possible.


6. Why Hardware Acceleration May Not Always Look Faster
-------------------------------------------------------

For very small toy RSA values, the accelerator may not always look dramatically
faster than software.

Reasons:

    1. Data must be converted into mbedTLS big-number format.
    2. Operands must be prepared before the hardware operation.
    3. The accelerator has setup overhead.
    4. Tiny 64-bit toy values are much smaller than real RSA values.
    5. Measurement noise can be large for short operations.

Hardware acceleration usually matters more for larger RSA-sized operands.
However, the toy workload is still useful because it teaches how to call,
measure, and validate the accelerator path.


7. Suggested Assignment Rule
----------------------------

Recommended rule:

    Students must keep both software and hardware paths correct.
    The software path is used to understand the algorithm.
    The hardware path is used to study ESP32 acceleration.

Students should not hard-code answers. They should compute:

    plaintext^e mod n

for every test case.


8. Suggested Scoring Addition
-----------------------------

Hardware accelerator usage can be graded like this:

    5 pts: hardware-accelerated path builds and runs
    5 pts: hardware path matches the software/reference result
    5 pts: student explains how ESP-IDF/mbedTLS uses ESP32 hardware MPI/RSA
    5 pts: student compares software and hardware timing clearly
    5 pts: student discusses why toy-value results may differ from real RSA

The instructor can either include these points in the main 100-point rubric or
make them a bonus section.


9. Recommended Student Explanation
----------------------------------

A good student report should say something like:

    The ESP32 has an RSA/MPI hardware accelerator for big-number modular
    arithmetic. I used mbedtls_mpi_exp_mod() with hardware MPI support enabled
    in ESP-IDF. I compared the result against the software baseline and checked
    that both returned the same ciphertext. I measured the software and hardware
    paths separately because accelerator setup and data conversion overhead can
    affect small toy benchmarks.


10. Key Takeaway
----------------

The ESP32 hardware accelerator is useful and should be part of this assignment,
but students still need to understand the software algorithm. The best learning
outcome comes from comparing both:

    software modular exponentiation
    vs.
    ESP32 hardware-accelerated modular exponentiation
